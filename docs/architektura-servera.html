Архитектура сервера

Блок-схемы ниже показывают, куда попадает запрос и какие файлы за что отвечают. Откройте этот файл в браузере (двойной щелчок или перетаскивание в Chrome/Edge).

Рисунок 1. Общий путь запроса

flowchart LR subgraph client["Клиент"] B[Браузер] end subgraph express["Express app.ts"] MW["Порядок: helmet → session → JSON → rate limit"] R[Роутеры по URL] end subgraph logic["Логика данных"] ST[data/store.ts] DB[(PostgreSQL)] MEM[mock + teacherMemory] end B --> MW --> R R --> ST ST -->|DATABASE_URL есть| DB ST -->|нет| MEM

Подпись: Сначала запрос проходит через настройки безопасности и сессии в app.ts, затем попадает в нужный роут. Данные для родителя почти всегда идут через store.ts, который выбирает БД или память.

Рисунок 2. Куда монтируются роутеры (важен порядок)

flowchart TB subgraph mounts["app.ts — префикс /api"] direction TB A1["/api/auth → authRouter"] A2["/api/profile → profileRouter"] A3["/api/parent → parentRouter"] A4["/api/teacher → requireTeacher → teacherRouter"] A5["/api/director → requireDirector → directorRouter"] A6["/api → requireParent → apiRouter родитель"] end A1 --> A2 --> A3 --> A4 --> A5 --> A6

Подпись: Узкие пути (auth, teacher…) регистрируются раньше, чем общий /api для родителя. Иначе родительский роутер перехватил бы чужие URL.

Рисунок 3. Вход в систему (слои)

flowchart TB subgraph http["HTTP"] RT["routes/auth.ts"] end subgraph svc["Сервис"] AS["auth/authService.ts
bcrypt, нормализация email"] end subgraph data["Данные пользователя"] AR["db/authRepository.ts
таблица users"] MD["auth/memDisplay.ts
демо без БД"] end RT --> AS AS -->|есть БД| AR AS -->|нет DATABASE_URL| MD

Подпись: Роут только принимает запрос; authService — правила входа; в БД логин ищется в authRepository, не в общем repository.ts.

Рисунок 4. Родитель: дневник и дети

flowchart LR API["routes/api.ts"] ST["data/store.ts"] REP["db/repository.ts"] MK["data/mock.ts"] API --> ST ST --> REP ST --> MK REP --> PG[(PostgreSQL
diary_*, students…)]

Подпись: api.ts отвечает за URL. Реальные SQL-запросы дневника и детей — в repository.ts. Оценки на карточке урока могут подстраиваться под ребёнка через lib/chemistryDayStudents.ts (внутри getDiary).

Рисунок 5. Учитель: БД или память

flowchart TB TR["routes/teacher.ts"] TR -->|DATABASE_URL| TREP["db/teacherRepository.ts
getClassDiary, roster…"] TR -->|нет БД| TM["data/teacherMemory.ts"] TREP --> PG2[(PostgreSQL
class_diary_*, school_classes)]

Подпись: HTTP-контракт всегда в teacher.ts. Источник данных переключается: PostgreSQL или демо в RAM (teacherMemory).

Рисунок 6. База данных: файлы и роли

flowchart TB subgraph sql["Схема DDL"] S1["schema.sql — базовая"] S2["schema_teacher.sql — классный дневник"] end subgraph pool["Доступ"] P["db/pool.ts → getPool()"] end subgraph repos["Репозитории запросов"] R1["repository.ts — родитель"] R2["teacherRepository.ts — класс"] R3["authRepository.ts — users"] R4["directorRepository.ts — директор"] R5["profileRepository.ts — профиль"] end sql --> repos P --> repos

Подпись: .sql — структура таблиц. pool — соединения. Каждый *Repository.ts — готовые запросы своей области. lessonRow.ts превращает строку SQL в тип DiaryLesson из types.ts.

Рисунок 7. Старт процесса и статика

flowchart TB SV["server.ts"] SV --> EC{{"ensureProfileDbCompatibility()
db/ensureCompatibility.ts"}} SV --> L["app.listen(PORT)"] APP["app.ts"] APP --> STC["express.static public/"] APP --> SPA["GET * → index.html SPA"] SV --> APP

Подпись: server.ts поднимает порт и при необходимости чинит колонки БД. Раздача сайта и SPA — внутри app.ts, не в server.ts.

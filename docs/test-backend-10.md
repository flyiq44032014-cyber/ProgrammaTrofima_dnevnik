# Бэкенд: слои, файлы и тест (30 вопросов)

Коротко для самопроверки в конце: ответы можно записать одной строкой `1B 2A …` и сверить с **ключом** внизу файла.

---

## 1. Зачем вообще «слои»

Слой — зона ответственности. Один файл не должен одновременно парсить HTTP, проверять пароль и строить длинный SQL: так проще читать код, тестировать и менять БД без трогания роутов.

Типичный поток **данных** (родитель/дневник при включённой БД):

`HTTP-запрос` → **роут** (`src/routes/…`) → **middleware** (сессия, роль) → **store** (`src/data/store.ts`) → **репозиторий** (`src/db/repository.ts` и узкие `*Repository.ts`) → **PostgreSQL** (через **pool**).

Для **входа и пароля** между роутом и `authRepository` стоит **сервис** (`src/auth/authService.ts`): там сценарий, `bcrypt`, вызовы репозитория.

---

## 2. Точка входа и «скелет» приложения

| Файл | Роль |
|------|------|
| `src/server.ts` | Запуск HTTP-сервера (порт, прослушивание). |
| `src/app.ts` | Создание приложения Express: middleware (сессия, helmet, лимиты), подключение **роутеров** из `src/routes/`. |

`app.ts` **не** описывает структуру таблиц и **не** выполняет произвольный SQL дневника — этим занимаются репозитории и DDL-файлы.

---

## 3. Роуты (`src/routes/`)

Здесь объявляются пути и привязка к обработчикам: например родительское API — `api.ts`, вход — `auth.ts`, профиль — `profile.ts`, учитель — `teacher.ts`, родительские маршруты — `parent.ts`, директор — `director.ts`.

Роут **не** должен содержать длинные SQL-строки и не обязан знать, mock сейчас или Postgres — для этого есть store и репозитории.

---

## 4. Middleware (`src/middleware/`)

- **`auth.ts`** — проверки вроде «есть ли сессия», «родитель ли», «учитель ли»: выполняются **до** основного кода обработчика.
- **`catchAsync.ts`** — обёртка над **async**-обработчиками: ловит ошибку, логирует, отдаёт JSON с единым форматом ошибки.

Это не слой «данных» и не замена `store.ts`.

---

## 5. Фасад данных: `src/data/store.ts`

`store` решает, **откуда** брать данные: реальная БД через репозитории или **in-memory** (`mock.ts`, память учителя и т.д., в зависимости от сценария).

После `store.get*` при БД обычно следует вызов методов **`src/db/repository.ts`** (или специализированного репозитория), а не повторный проход через `requireParent`.

---

## 6. Сервисы (сценарии, не SQL)

| Файл | Роль |
|------|------|
| `src/auth/authService.ts` | Логин/регистрация: `bcrypt.compare` / хеширование, оркестрация вызовов `authRepository`. |
| `src/profile/service.ts` | Сбор **единого JSON профиля** (ФИО, дети, телефон, классы учителя и пр.) с ветвлением БД / память. |

**`bcrypt.compare`** и «что делать, если пароль неверный» — зона сервиса. Репозиторий **отдаёт** из БД `password_hash`, а не сравнивает открытый пароль внутри SQL-слоя осмысленно как бизнес-логика.

---

## 7. Репозитории и SQL (`src/db/`)

- **`repository.ts`** — основной SQL для дневника, детей, расписания и связанных таблиц (по задумке проекта).
- **`authRepository.ts`** — таблица **`users`**: поиск по email, вставка пользователя, чтение хеша и т.д.
- **`teacherRepository.ts`**, **`profileRepository.ts`**, **`directorRepository.ts`** — узкие модули под свои сущности.

Репозиторий использует **пул** (`pool.ts`), а не «хранит дневник в себе».

---

## 8. Инфраструктура БД

| Файл | Роль |
|------|------|
| `src/db/pool.ts` | `getPool()` — **пул подключений** к PostgreSQL. Это не бизнес-данные и не роуты. |
| `src/db/schema.sql`, `schema_teacher.sql`, др. | **DDL**: `CREATE TABLE`, индексы — описание структуры БД, не выполнение `bcrypt` и не HTTP. |

---

## 9. Mock и типы

- **`src/data/mock.ts`** и смежные in-memory источники — данные без Postgres для разработки/демо (в связке со `store`).
- **`src/types.ts`** — TypeScript-типы и интерфейсы, **не** runtime-логика и не SQL.

---

## 10. Предметная логика (`src/lib/`)

Например **`chemistryDayStudents.ts`** — правила/преобразования вокруг уроков и списков: это ближе к **доменной** логике, чем к слою HTTP. Роут может **вызывать** такие функции, но сам файл не является «таблицей маршрутов».
